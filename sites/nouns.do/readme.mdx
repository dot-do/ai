# nouns.do

**Semantic entity types and noun vocabularies for Business-as-Code applications**

nouns.do provides a comprehensive, AI-native vocabulary of entity types based on Schema.org's Thing hierarchy. It enables developers and AI agents to work with strongly-typed business entities using semantic patterns that machines and humans can understand.

## Overview

In Business-as-Code, **nouns** represent entities - the people, places, things, concepts, and ideas that make up your business domain. nouns.do provides:

- **817+ Entity Types** from Schema.org (Person, Organization, Product, Service, etc.)
- **Semantic Type Hierarchies** following the Thing → SubType pattern
- **Type-Safe TypeScript Definitions** for all entity types
- **AI-Native Patterns** using `$.Noun` semantic proxy
- **Linked Data Support** with `$id` and `$type` properties
- **Extensible Vocabularies** for domain-specific entity types

Install the SDK:

```bash

```

```typescript

// Create entity instances using noun types

```

## Core Concepts

### What Are Nouns?

In Business-as-Code, **nouns** are entity types that represent real-world concepts:

- **People**: Person, Employee, Customer, User
- **Organizations**: Organization, Corporation, LocalBusiness, EducationalOrganization
- **Places**: Place, PostalAddress, City, Country
- **Things**: Product, Service, Vehicle, Book, Article
- **Creative Works**: Article, BlogPosting, Comment, Review
- **Events**: Event, BusinessEvent, SocialEvent, PublicationEvent
- **Actions**: Action, TradeAction, ConsumeAction, CommunicateAction
- **Intangibles**: JobPosting, Offer, Order, Invoice, Ticket

### The Thing Hierarchy

All Schema.org types inherit from the base `Thing` type:

```typescript
// Base Thing type
interface Thing {
  $id?: string // Unique identifier
  $type: string // Type name
  name?: string // Name of the thing
  description?: string // Description
  url?: string // URL
  image?: string // Image URL
  identifier?: string // Alternative identifier
}

// Example: Person extends Thing
interface Person extends Thing {
  $type: 'Person'
  givenName?: string
  familyName?: string
  email?: string
  telephone?: string
  jobTitle?: string
  worksFor?: Organization
}

// Example: Organization extends Thing
interface Organization extends Thing {
  $type: 'Organization'
  legalName?: string
  taxID?: string
  industry?: string
  foundingDate?: string
  numberOfEmployees?: number
  address?: PostalAddress
}
```

### Semantic Noun Patterns

nouns.do uses the `$` semantic proxy for type-safe entity access:

```typescript
import $ from 'sdk.do'

// Access noun types
$.Person // PersonType
$.Organization // OrganizationType
$.Product // ProductType
$.Place // PlaceType
$.Event // EventType
$.Action // ActionType

// Use in database operations
const person = await db.create($.Person, { name: 'John' })
const products = await db.list($.Product, { limit: 10 })

// Use in relationships
await db.relate(person, $.worksFor, organization)
await db.relate(organization, $.owns, brand)
```

## Working with Noun Types

### Type Safety with TypeScript

nouns.do provides full TypeScript support:

```typescript
import type { Person, Organization, Product } from 'schema.org.ai'
import $, { db } from 'sdk.do'

// Type-safe entity creation
const person: Person = await db.create($.Person, {
  $type: 'Person',
  name: 'Jane Smith',
  email: 'jane@example.com',
  jobTitle: 'CTO',
})

// Type-safe queries
const people: Person[] = await db.list($.Person, {
  where: { jobTitle: 'CTO' },
})

// TypeScript ensures property correctness
person.name // ✓ Valid
person.jobTitle // ✓ Valid
person.invalidProp // ✗ TypeScript error
```

### AI-Generated Entities

Use AI to generate entities that conform to noun types:

```typescript
import { ai } from 'sdk.do'

// Generate a person entity
const person = await ai.generate('Create a software engineer profile for Sarah Chen', { schema: $.Person })
// Returns: { $type: 'Person', name: 'Sarah Chen', jobTitle: 'Software Engineer', ... }

// Generate an organization
const org = await ai.generate('Create a profile for a sustainable energy startup', { schema: $.Organization })
// Returns: { $type: 'Organization', name: '...', industry: 'Renewable Energy', ... }

// Generate products
const products = await ai.batch([
  { prompt: 'Create a smart home device', schema: $.Product },
  { prompt: 'Create a fitness tracker', schema: $.Product },
  { prompt: 'Create a wireless speaker', schema: $.Product },
])
```

### Validating Entities

Ensure entities conform to their noun types:

```typescript
import { validate } from 'sdk.do'

const person = {
  $type: 'Person',
  name: 'John Doe',
  email: 'invalid-email', // Invalid format
}

// Validate against schema
const validation = validate(person, $.Person)

if (!validation.valid) {
  console.error('Validation errors:', validation.errors)
  // errors: [{ path: 'email', message: 'Invalid email format' }]
}
```

### Querying by Type

Query entities by their noun types:

```typescript
// List all entities of a type
const people = await db.list($.Person)
const organizations = await db.list($.Organization)
const products = await db.list($.Product)

// Filter by type properties
const engineers = await db.list($.Person, {
  where: { jobTitle: { $contains: 'Engineer' } },
})

const techCompanies = await db.list($.Organization, {
  where: { industry: 'Technology' },
})

const affordableProducts = await db.list($.Product, {
  where: { price: { $lt: 100 } },
})

// Get related entities by type
const employer = await db.related(person, $.worksFor, $.Organization)
const employees = await db.related(organization, $.employs, $.Person)
```

## Advanced Patterns

### Custom Noun Types

Extend Schema.org types with domain-specific nouns:

```typescript
import $ from 'sdk.do'

// Define custom noun type
interface Lead extends $.Person {
  $type: 'Lead'
  leadSource: string
  leadScore: number
  leadStatus: 'New' | 'Qualified' | 'Converted' | 'Lost'
  assignedTo?: string
}

// Register custom type
$.register('Lead', {
  baseType: $.Person,
  properties: {
    leadSource: { type: 'string' },
    leadScore: { type: 'number' },
    leadStatus: { type: 'string', enum: ['New', 'Qualified', 'Converted', 'Lost'] },
    assignedTo: { type: 'string', optional: true },
  },
})

// Use custom type
const lead: Lead = await db.create($.Lead, {
  $type: 'Lead',
  name: 'Potential Customer',
  email: 'customer@example.com',
  leadSource: 'Website',
  leadScore: 85,
  leadStatus: 'Qualified',
})
```

### Noun Composition

Combine multiple noun types:

```typescript
// Product with organization brand
const product = await db.create($.Product, {

  },

    $type: 'Organization',
    name: 'Acme Corporation',

      $type: 'PostalAddress',
      addressLocality: 'San Francisco',
    },
  },
  offers: {
    $type: 'Offer',

```

### Polymorphic Queries

Query across type hierarchies:

```typescript
// Query all organizations (includes Corporation, LocalBusiness, etc.)
const allOrgs = await db.list($.Organization)

// Query specific subtypes
const corporations = await db.list($.Organization, {
  where: { $type: 'Corporation' },
})

const localBusinesses = await db.list($.Organization, {
  where: { $type: 'LocalBusiness' },
})

// Type narrowing
function processEntity(entity: $.Thing) {
  if (entity.$type === 'Person') {
    console.log(`Person: ${entity.name}`)
  } else if (entity.$type === 'Organization') {
    console.log(`Organization: ${entity.legalName || entity.name}`)
  }
}
```

## Integration with Platform

### With verbs.do

Combine nouns with verbs for actions:

```typescript
import $ from 'sdk.do'

// Nouns define entities
const sender = await db.get($.Person, senderId)
const recipient = await db.get($.Person, recipientId)

// Verbs define actions
await $.send(sender, $.Message, recipient, {
  subject: 'Hello',

```

### With resources.do

Nouns define types, resources are instances:

```typescript
// Noun: Entity type
$.Person // Type definition

// Resource: Entity instance
const person = await db.create($.Person, {
  name: 'John',
}) // Actual person resource
```

### With graphdl.do

Build semantic graphs with nouns:

```typescript
import { graph } from 'graphdl.do'

// Define graph with noun types
const businessGraph = graph`
  ${person: $.Person} -[${$.worksFor}]-> ${org: $.Organization}
  ${org} -[${$.owns}]-> ${brand: $.Brand}
  ${brand} -[${$.manufactures}]-> ${product: $.Product}
`

// Query graph
const results = await businessGraph.query({
  person: { name: 'John Doe' }

```

## Best Practices

### 1. Use Semantic Types

Always use `$.NounType` for type safety:

```typescript
// Good - semantic and type-safe
const person = await db.create($.Person, data)

// Avoid - string-based, error-prone
const person = await db.create('Person', data)
```

### 2. Include $type Property

Always specify the `$type` property:

```typescript
// Good - explicit type
const person = await db.create($.Person, {
  $type: 'Person',
  name: 'Jane',
})

// Risky - implicit type
const person = await db.create($.Person, {
  name: 'Jane',
})
```

### 3. Follow Type Hierarchies

Respect the Schema.org hierarchy:

```typescript
// Good - proper hierarchy
interface Employee extends Person {
  $type: 'Employee'
  employeeId: string
}

// Avoid - breaks hierarchy
interface Employee {
  $type: 'Employee'
  name: string // Duplicates Person properties
}
```

### 4. Use Linked Data

Leverage `$id` for entity references:

```typescript
// Good - linked data
const person = {
  $id: 'https://example.com/people/jane',
  $type: 'Person',
  name: 'Jane',
  worksFor: {
    $id: 'https://example.com/orgs/acme',
    $type: 'Organization',
  },
}

// Avoid - unlinked data
const person = {
  name: 'Jane',
  employer: 'Acme', // No semantic relationship
}
```

### 5. Validate Entity Data

Always validate entities before storage:

```typescript
import { validate } from 'sdk.do'

const person = { $type: 'Person', name: 'Jane', email: 'jane@example.com' }

if (validate(person, $.Person)) {
  await db.create($.Person, person)
} else {
  throw new Error('Invalid person data')
}
```
