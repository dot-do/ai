# Event Naming Conventions

Comprehensive guide to the `Object.Action` naming pattern for semantic business events.

## Overview

Events in the `.do` platform follow the `Object.Action` naming convention, where:

- **Object**: The business entity or domain object (Order, Payment, User, Product, etc.)
- **Action**: Past tense verb describing what happened (created, updated, deleted, shipped, etc.)

This pattern is:

- **Human-readable**: Easy to understand at a glance
- **AI-understandable**: Semantic structure for AI processing
- **Consistent**: Follows predictable patterns across all domains
- **Searchable**: Easy to filter and query events

## Basic Format

```typescript
'{Object}.{Action}'
```

### Examples

```typescript
'Order.created' // An order was created
'Payment.completed' // A payment was completed
'User.registered' // A user registered
'Product.updated' // A product was updated
'Inventory.reserved' // Inventory was reserved
'Shipment.delivered' // A shipment was delivered
```

## Object Names

### Business Entities

Use singular nouns for business entities:

```typescript
'Order.created' // Not Orders.created
'Payment.completed' // Not Payments.completed
'User.registered' // Not Users.registered
'Product.updated' // Not Products.updated
```

### Composite Objects

Use PascalCase for multi-word objects:

```typescript
'OrderItem.added'
'ShoppingCart.updated'
'PaymentMethod.verified'
'CustomerAddress.changed'
'ProductCategory.created'
```

## Action Names

### Use Past Tense

Always use past tense verbs to indicate the action has occurred:

```typescript
// Good
'Order.created'
'Payment.completed'
'User.verified'

// Bad
'Order.create'
'Payment.complete'
'User.verify'
```

## Special Patterns

### Status Change Events

For explicit status changes:

```typescript
'{Object}.statusChanged'

await send('Order.statusChanged', {
  orderId: '12345',
  oldStatus: 'pending',
  newStatus: 'shipped',
})
```

### Error Events

For error conditions:

```typescript
'{Object}.{Action}Failed'

'Order.creationFailed'
'Payment.processingFailed'
'Email.sendingFailed'
'Import.failed'
```

### Notification Events

For notifications about entities:

```typescript
'{Entity}.notified'

'Customer.notified'
'Admin.notified'
'User.alerted'
```

### Aggregation Events

For aggregated or summary events:

```typescript
'{Object}.{Period}Summary'

'Sales.dailySummary'
'User.weeklySummary'
'Revenue.monthlySummary'
```

## Anti-Patterns

### Avoid Ambiguous Names

```typescript
// Bad - Unclear what changed
'Order.changed'
'User.modified'

// Good - Specific action
'Order.statusChanged'
'User.emailUpdated'
```

### Avoid Abbreviations

```typescript
// Bad
'Ord.crtd'
'Pmt.cmpltd'

// Good
'Order.created'
'Payment.completed'
```

### Avoid Generic Actions

```typescript
// Bad - Too generic
'Order.processed'
'User.handled'

// Good - Specific action
'Order.fulfilled'
'User.verified'
```

### Avoid Present Tense

```typescript
// Bad
'Order.creating'
'Payment.processing'

// Good
'Order.created'
'Payment.processed'
```

### Avoid Underscores or Hyphens

```typescript
// Bad
'order_created'
'payment-completed'
'user.email_verified'

// Good
'Order.created'
'Payment.completed'
'User.emailVerified'
```

## Naming Checklist

Use this checklist when creating event names:

- [ ] Object is a singular noun (Order, not Orders)
- [ ] Object uses PascalCase (OrderItem, not order_item)
- [ ] Action is past tense (created, not create)
- [ ] Action is specific (validated, not processed)
- [ ] Name follows Object.Action format
- [ ] Name is human-readable
- [ ] Name is self-documenting
- [ ] Name is consistent with existing events
- [ ] No abbreviations or acronyms
- [ ] No underscores or hyphens in action

## Semantic Integration

### Using with $ Proxy

The `$` proxy provides semantic context:

```typescript
import $, { send } from 'sdk.do'

// Semantic event names
await send($.Order.created, orderData)
await send($.Payment.completed, paymentData)
await send($.User.registered, userData)
```

### Schema.org Integration

Events can reference Schema.org types:

```typescript
import { send } from 'sdk.do'

await send('Order.created', {
  $type: 'Order',
  $id: 'order-12345',
  orderNumber: '12345',
  customer: {
    $type: 'Organization',
    name: 'Acme Corp',
  },
  orderedItem: [
    {
      $type: 'Product',
      name: 'Widget Pro',
      sku: 'WID-PRO-001',
    },
  ],
})
```

## Documentation

Document your events in code:

```typescript
/**
 * Published when an order is successfully created.
 *
 * @event Order.created
 * @property {string} id - Order ID
 * @property {object} customer - Customer information
 * @property {array} items - Order items
 * @property {number} total - Order total
 * @property {string} currency - Currency code
 * @property {Date} createdAt - Creation timestamp
 */
await send('Order.created', {
  id: '12345',
  customer: { id: 'cust-1', name: 'Acme Corp' },
  items: [{ product: 'widget', quantity: 10, price: 99.99 }],
  total: 999.9,
  currency: 'USD',
  createdAt: new Date(),
})
```

## Type Definitions

Define TypeScript types for your events:

```typescript
// Event type definition
interface OrderCreatedEvent {
  type: 'Order.created'
  data: {
    id: string
    customer: {
      id: string
      name: string
      email: string
    }
    items: Array<{
      product: string
      quantity: number
      price: number
    }>
    total: number
    currency: string
    createdAt: Date
  }
  metadata?: EventMetadata
}

// Usage with type safety
const event: OrderCreatedEvent = {
  type: 'Order.created',
  data: {
    id: '12345',
    customer: { id: 'cust-1', name: 'Acme', email: 'orders@acme.com' },
    items: [{ product: 'widget', quantity: 10, price: 99.99 }],
    total: 999.9,
    currency: 'USD',
    createdAt: new Date(),
  },
}
```

## Versioning Events

Include version in metadata for schema evolution:

```typescript
await send('Order.created', orderData, {
  version: 2,
  schemaUrl: 'https://schemas.example.com/Order.created/v2',
})
```

## Best Practices

1. **Be Consistent**: Follow the same pattern across all events
2. **Be Specific**: Use precise action names (validated vs processed)
3. **Be Semantic**: Names should convey meaning without context
4. **Be Past Tense**: Events describe what happened, not what will happen
5. **Be Singular**: Use singular nouns (Order, not Orders)
6. **Be PascalCase**: Use PascalCase for object and action
7. **Document Events**: Maintain a registry of all events
8. **Type Events**: Define TypeScript types for events
9. **Version Events**: Include version in metadata when needed
10. **Review Events**: Regularly review and refactor event names
